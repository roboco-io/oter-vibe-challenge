# PRD — oter Vibe Coding 채용 과제

## 배경

공터영어(oter)는 레거시 PHP로 작성된 서비스를 클라우드 네이티브 환경으로 이전하면서, 동시에 장기적으로는 자유로운 스택으로 재구축하는 전환기를 지나고 있습니다. 이 과제는 그 현실을 축소한 시뮬레이션입니다: 지원자에게 동작하는 레거시 PHP RealWorld 앱(Conduit 클론)을 제공하고, (1) 있는 그대로 NHN Cloud NCS에 컨테이너로 배포(마이그레이션)한 뒤, (2) 자기 선택 스택으로 재구현하고, (3) RealWorld 공식 E2E 테스트 스위트로 두 결과물이 동등하게 동작함을 증명하도록 요구합니다.

평가 대상은 결과물의 완성도만이 아니라, **AI를 도구로 활용하며 설계를 통제하는 능력**과 **일주일간의 성장 궤적**입니다.

## Must-have (필수)

| ID | 항목 | 설명 |
|---|---|---|
| M1 | PHP 앱 NCS 컨테이너 배포 (마이그레이션) | 제공된 레거시 PHP RealWorld 앱을 수정 없이(또는 최소 수정으로) 컨테이너화하여 NHN Cloud NCS에 배포한다. |
| M2 | 공식 E2E(RealWorld API) 그린 (안전망) | RealWorld 공식 API 통합 테스트 스위트를 M1 배포본에 대해 실행하여 전부 통과시킨다. 이후 재구현본의 동등성 판단 기준이 된다. |
| M3 | 선택 스택 재구현 + 동일 E2E 통과 (동등성) | 자기 선택 언어·프레임워크로 API를 재구현하고, M2와 동일한 공식 E2E 스위트를 통과시켜 동등성을 증명한다. |
| M4 | 스택 선택 ADR + Terraform IaC | 재구현 스택 선택 근거를 ADR로 문서화하고, NCS 인프라를 Terraform으로 코드화한다. |
| M5 | GitHub Actions CI (빌드 → NCR → NCS) | 빌드, NHN Cloud Container Registry(NCR) 푸시, NCS 배포까지 이어지는 CI 파이프라인을 GitHub Actions로 구성한다. |
| M6 | 데일리 push + Retrobot 회고 + tokenhabit 리포트 | 매일 커밋을 push하고, Retrobot으로 KPT 회고를 남기며, tokenhabit으로 토큰 사용 리포트를 남긴다. |

## Stretch (선택)

완수 시 가산점 대상이며, Must-have를 희생하면서까지 시도할 필요는 없습니다.

- 관측성 (로그·메트릭 수집 및 대시보드)
- 무중단 배포·오토스케일링
- 비용 최적화 (인스턴스 사이징, 스케줄링 등)
- 프론트엔드 연동 (RealWorld 공식 프론트엔드 클라이언트 붙이기)
- 부하 테스트

## 고정 스택 / 자유 스택

- **고정**: NHN Cloud, NCS(Container Service), Terraform, GitHub Actions — 인프라·배포·CI 도구는 변경 불가.
- **자유**: 애플리케이션 언어·프레임워크는 지원자가 자유롭게 선택한다. 단, 선택 근거를 ADR로 반드시 정당화해야 한다.

## 예산

- 지원자가 자비로 NHN Cloud 비용을 부담하며, **제출 완료 시 5만원을 보전**합니다.
- 소형 인스턴스 사용과 작업 종료 후 teardown(자원 정리)을 통해 예산 범위 내에서 운영하는 것 자체가 **비용 인식 능력을 보는 시험**입니다.

## 성장 3종 측정

지원자의 일주일간 성장을 다음 세 가지로 측정합니다.

1. **Retrobot 회고 궤적**: 매일 KPT(Keep-Problem-Try) 형식 회고를 남기고, 회고 내용이 시간이 지날수록 구체화·심화되는지를 본다.
2. **tokenhabit 추세**: 초반 대비 후반의 토큰 사용 패턴이 개선되는지(불필요한 재시도 감소, 프롬프트 정교화 등)를 본다.
3. **ADR·발표 자기성찰**: ADR과 최종 발표에서 자신의 선택을 되돌아보고 한계·트레이드오프를 스스로 인식하는지를 본다.

## 불합격 기준

다음에 해당하면 다른 항목 점수와 무관하게 불합격 처리합니다.

- E2E 테스트 없이 재구현만 제출한 경우
- 시크릿(API 키, 자격 증명 등)을 커밋한 경우
- 배포가 실제로 동작하지 않는 경우
- AI 사용 사실을 은폐한 경우

## 평가 원칙

**완성도보다 설계 의도와 성장을 봅니다.** 부분 구현이라도 설계 근거와 회고가 충실하다면 그만큼 보완하여 평가합니다. 반대로 완성도가 높아도 설계 의도가 불분명하거나 성장의 흔적이 없다면 낮게 평가할 수 있습니다.
